# RTXNS 夏季 Slang 与神经渲染 12 周计划

## 1. 文档目的

本文档用于指导 `D:\xmd\RTXNS` 在 2026 年 6 月 1 日至 2026 年 8 月 23 日这 12 周内的阶段推进、每周汇报与阶段性交付；2026 年 8 月 24 日至 2026 年 8 月 31 日作为机动收口周，用于补实验、补文档和应对老师反馈。

这份计划是对 [DonutPython_12Week_Plan.md](/D:/xmd/RTXNS/docs/DonutPython_12Week_Plan.md) 的自然延续，但重点已经从“把 Donut Python 后端做出来”，转向：

- 把现有 RTXNS 渲染器做成更适合性能分析、神经渲染和后续论文抽象的实验平台；
- 吸收 2026 年 6-8 月在 UE / Slang / 渲染优化方向上的实习经验，反哺回 `D:\xmd\RTXNS`；
- 围绕 Slang 接入、神经渲染原型和性能优化，持续产出适合每周向老师汇报的阶段成果。

因此，这份计划的目标不是把所有神经渲染方法一次性做完，而是把后续最值得深入的一条线跑通，并把工程、实验和论文叙事三件事同步搭起来。

## 2. 当前基线与正确定位

### 2.1 当前已经具备的基础

当前仓库已经不是“从零开始”的状态，而是具备了继续往神经渲染方向推进的基础：

- 已经有 `C++ -> pybind11 -> Python` 的稳定调用链；
- 已经有基于 Donut / Vulkan 的无头离屏渲染流程；
- 已经可以融入 Genesis 风格的渲染调用与场景更新管线；
- 已经有面向 Python 的对象化封装、scene update 规划与若干 smoke 验证；
- 已经具备继续扩展多缓冲输出、Slang pass 和性能测试的工程落脚点。

这意味着，接下来的重点不应该再停留在“再证明一次能渲染出图片”，而应该转向：

- 性能画像与瓶颈定位；
- 神经渲染所需数据出口和中间表示；
- Slang 最小执行通路；
- 可复用、可汇报、可写论文的实验闭环。

### 2.2 这三个月真正要做什么

这三个月在老师那里更合理的表述，不是“去做一个完整的新引擎”，而是：

1. 以 `RTXNS` 为实验室渲染器平台，持续做性能分析与渲染优化。
2. 以 Slang 为核心切入点，为神经渲染能力接入打基础。
3. 从 `RTXNTC`、`D:\xmd\ls\RTXNS`、NeuMIP、神经光照缓存等思路中选择最合适的一条主线做原型。
4. 把过程中得到的资源组织、运行时结构、性能收益和实验结论沉淀成后续毕业论文的候选创新点。

### 2.3 这三个月不应被定义成什么

为了避免范围漂移，需要明确这三个月不等于：

- 不等于把 UE5 的整套渲染管线“搬进” `RTXNS`；
- 不等于一次性同时做完 `NTC + NeuMIP + 光照缓存 + 训练系统`；
- 不等于先承诺某篇论文级结果，再倒推工程；
- 不等于为了做演示而牺牲后续可维护性和实验可重复性。

更合适的定义是：

- 6 月偏“性能分析、渲染数据出口、工程可观测性”；
- 7 月偏“Slang 通路和最小神经 pass”；
- 8 月偏“性能优化、实验收口、论文抽象”。

## 3. 12 周后的交付定义

12 周结束时，理想交付不应只是几个零散 demo，而应至少满足下面这些条件：

1. `RTXNS` 有一套可复用的性能基线与 benchmark 口径。
2. 渲染器能稳定导出神经渲染实验所需的最小多缓冲数据，而不只是一张 RGBA。
3. 仓库内形成一条可复用的 Slang 最小执行通路，能够承接后处理或神经 pass。
4. 至少有一条神经渲染主线完成“最小可运行原型”。
   - 可选方向包括：NTC 风格材质压缩/解码、NeuMIP 风格神经材质推理、神经光照缓存/重光照。
5. 至少形成一个可以向老师明确讲清楚的性能优化故事。
   - 例如：某类资源复用、某条场景更新路径、某个 Slang pass、某组 neural features 的性能收益。
6. 至少形成一版可写进毕业论文的研究叙事草稿。
   - 包括问题定义、系统设计、实验口径、阶段性结果与局限。

## 4. 总体推进策略

整个 12 周建议遵循“先把平台搭对，再把算法接上，最后再做论文抽象”的顺序推进：

### 4.1 先做性能和数据，不先许诺算法结果

如果没有稳定 benchmark、可观测指标和多缓冲输出，那么即使做出一个神经 pass，也很难证明它值得继续做。

### 4.2 先做 Slang 最小通路，不先做大而全框架

优先目标是打通“Donut/RTXNS 输出 -> Slang pass -> 新结果输出”这一条最短路径，而不是一开始就做完整训练系统。

### 4.3 先做最小原型，不同时押注多个研究方向

`NTC`、`NeuMIP`、神经光照缓存都可以作为灵感来源，但这三个月更适合：

- 先选一个主线；
- 另保留一个备选线；
- 用实验结果而不是预设偏好决定后续方向。

### 4.4 让实习经验服务于 RTXNS，而不是相反

在 UE 实习里学到的性能分析、shader 组织、资源管理和 Slang 接入经验，应该反哺成 `RTXNS` 的：

- profiling 方法；
- runtime 结构；
- 资源生命周期规范；
- 性能优化思路；
- 与论文相关的工程论证。

## 5. 候选技术主线与选题原则

### 5.1 候选主线 A：NTC 风格神经纹理压缩

这一方向更适合围绕：

- 显存 / 存储占用；
- 材质纹理组织方式；
- on-sample / on-load 风格的取舍；
- 纹理重建质量与带宽成本。

这条线更偏“材质表示与资源优化”。

### 5.2 候选主线 B：NeuMIP 风格神经材质推理

这一方向更适合围绕：

- 神经纹理 + 小型网络的材质表示；
- Slang 下的推理 pass；
- 材质细节重建、多尺度表示与视角相关效果；
- 与已有 `D:\xmd\ls\RTXNS` 的经验形成连续性。

这条线更偏“神经材质表示与推理系统”。

### 5.3 候选主线 C：神经光照缓存 / 重光照

这一方向更适合围绕：

- 神经缓存中间表示；
- 间接光或重光照的近似；
- 训练数据与监督信号导出；
- 动态场景下的更新与预算控制。

这条线更偏“光照与缓存系统”。

### 5.4 选题原则

7 月中旬前后建议按下面标准决定主线：

1. 能否自然挂接到当前 `RTXNS` 渲染流程。
2. 能否在 1-2 周内做出最小可运行版本。
3. 能否产生可量化的性能、质量或内存指标。
4. 能否形成一段容易向老师讲清楚的系统故事。
5. 能否为毕业论文留下后续可扩展空间。

## 6. 周报模板

建议从这三个月开始，每周汇报尽量固定成同一结构：

### 周报结构

- 本周主线
- 本周完成
- 本周验证
- 当前瓶颈 / 风险
- 下周计划
- 对论文或长期方向的意义

### 建议补充的一项

为了把实习和实验室工作自然串起来，建议每周再多补一项：

- 本周从 UE / Slang / 渲染优化实践中迁移到 RTXNS 的方法或认识

### 示例

- 本周主线：完成 `RTXNS` 的性能基线场景与 profiling 口径冻结。
- 本周完成：补齐 benchmark 场景、增加计时与多帧统计、整理 scene update 热点路径。
- 本周验证：对固定场景完成 3 轮重复运行，得到可复现的平均耗时与方差。
- 当前风险：虽然知道热点在哪，但暂时还没有适合神经渲染实验的数据出口。
- 下周计划：补 depth / normal / material 参数等多缓冲输出，并评估 Slang pass 接入点。
- 对论文或长期方向的意义：后续所有“优化有效”或“神经方案值得做”的论断，都要以这套基线为依据。

## 7. 面向代码 Agent 的执行规则

为了方便后续代码 Agent 根据计划落地实验，每周任务建议遵循下面的规则：

### 任务模板

- 目标
- 范围
- 预期修改模块
- 验收方式
- 允许调整的空间

### Agent 执行规则

1. 每周只保留一个主主题，不把过多研究方向混在同一周。
2. 每周至少落下一项可 review 的产物。
   - 可以是代码、文档、benchmark、图表、实验脚本或报告。
3. 每周至少有一项可以复现的验证。
   - 可以是图片、耗时数据、质量指标、日志、单测或 smoke。
4. 如果算法方向暂时跑不通，应优先退回去补平台能力，而不是空转。
   - 例如补 profiling、buffer 导出、资源复用、文档、benchmark。
5. 不把“可能的论文点”写成“已经成立的创新点”。
   - 只有在有数据、有对比、有边界说明时，才升级为正式结论。

### 每周任务完成判定

一周工作只有同时满足下面 4 条，才算真正适合用于老师汇报：

1. 仓库里有实物产出。
2. 结果可以被别人复现或至少被自己重复验证。
3. 能用 3-5 句话讲清楚“为什么这周做它”。
4. 能明确说明它与后续 Slang / 神经渲染 / 论文主线的关系。

- 把仓库状态和文档状态整理干净；
- 为 9 月后的连续推进留出缓冲。

## 9. 每月汇报重点

### 2026 年 6 月汇报重点

- 已经从“能跑的渲染器原型”进入“可测量、可分析、可继续扩展的实验平台”阶段；
- 性能基线、benchmark 口径和 profiling 方法已经逐步成型；
- 神经渲染所需的数据出口正在补齐，后续 Slang 接入有明确落点。

### 2026 年 7 月汇报重点

- `RTXNS` 里已经开始形成 Slang 的独立执行通路；
- 已经从纯 PBR / 纯后端工程，推进到最小神经 pass 或最小神经材质能力原型；
- 主线方向已经在 `NTC / NeuMIP / 光照缓存` 中初步收敛。

### 2026 年 8 月汇报重点

- 已经有可以讲清楚的性能优化故事和实验对比口径；
- 已经能把系统设计、实验结果和论文方向连成一条线；
- 已经为后续继续做大论文或更正式项目打下第一版工程与实验基础。

## 10. 这 12 周内不应过度扩展的方向

为了保证这三个月的汇报质量，建议主动避免下面这些扩展：

- 直接重写完整训练框架；
- 同时并行推进多个大型神经渲染课题；
- 过早做复杂 UI、编辑器或资产系统；
- 在没有稳定数据前，过早包装成“论文级结果”；
- 把主要时间消耗在与主线关系弱的零散美化或重构上。

这些方向不是永远不做，而是不应挤占这 12 周的主线窗口。

## 11. 可写进毕业论文的创新点候选

下面这些内容可以作为候选，而不是提前宣称已经成立的创新点：

1. 面向仿真器离屏渲染场景的 Slang 神经 pass 接入结构。
2. 结合 Python 调度、C++ 运行时与 GPU pass 的轻量神经渲染实验平台设计。
3. 针对神经材质 / 神经纹理 / 神经缓存的资源组织与性能优化策略。
4. 在 `RTXNS` 这种实验室渲染器中，把性能测量、数据导出和神经渲染原型统一到同一套运行时的工程方法。

如果后续数据足够扎实，这些候选点可以进一步收敛成：

- 系统型论文叙事；
- 方法 + 工程结合型毕业论文章节；
- 或者一条面向后续更长周期课题的前期技术积累线。

## 12. 每周必须留下的交付痕迹

每周结束时，至少应留下下面 5 类痕迹中的大部分：

- 一份简短书面总结；
- 一份代码或文档 diff；
- 一项可重复验证结果；
- 一个明确的下周目标；
- 一段对当前风险与边界的说明。

如果某一周非常忙，但最后什么痕迹都没有，那一周就很难支撑老师汇报，也不利于后续代码 Agent 接手继续推进。
